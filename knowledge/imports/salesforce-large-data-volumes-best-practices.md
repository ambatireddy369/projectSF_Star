## Introduction

### Who Should Read This

This paper is for experienced application architects who work with Salesforce deployments that contain large data volumes.

A "large data volume" is an imprecise, elastic term. If your deployment has tens of thousands of users, tens of millions of records, or hundreds of gigabytes of total record storage, you have a large data volume. Even if you work with smaller deployments, you can still learn something from these best practices.

### Overview

Salesforce enables customers to easily scale up their applications from small to large amounts of data. This scaling usually happens automatically, but as data sets get larger, the time required for certain operations grows. The ways in which architects design and configure data structures and operations can increase or decrease those operation times by several orders of magnitude.

The main processes affected by differing architectures and configurations are:

- Loading or updating of large numbers of records, either directly or with integrations
- Extraction of data through reports and queries, or through views

The strategies for optimizing those main processes are:

- Following industry-standard practices for accommodating schema changes and operations in database-enabled applications
- Deferring or bypassing business rule and sharing processing
- Choosing the most efficient operation for accomplishing a task

### What's in This Paper

- Techniques for improving the performance of applications with large data volumes
- Salesforce mechanisms and implementations that affect performance in less-than-obvious ways
- Salesforce mechanisms designed to support the performance of systems with large data volumes

### Salesforce Big Objects

Salesforce provides big data technology called Big Objects. A big object stores and manages massive amounts of data on the Salesforce platform. You can archive data from other objects or bring massive datasets from outside systems into a big object to get a full view of your customers. A big object provides consistent performance, whether you have 1 million records, 100 million, or even 1 billion.

This paper focuses on optimizing large data volumes stored in standard and custom objects, not big objects. For optimal performance and a sustainable long-term storage solution for even larger data sets, use Bulk API or Batch Apex to move your data into big objects.

## Underlying Concepts

### Multitenancy and Metadata Overview

Multitenancy is a means of providing a single application to multiple organizations from a single hardware-software stack. Instead of providing a complete set of hardware and software resources to each organization, Salesforce inserts a layer of software between the single instance and each organization's deployment.

When organizations create custom objects, the platform tracks metadata about the objects and their fields, relationships, and other object definition characteristics. Salesforce stores the application data for all virtual tables in a few large database tables, which are partitioned by tenant and serve as heap storage.

Instead of attempting to manage a vast, ever-changing set of actual database structures for each application and tenant, the platform storage model manages virtual database structures using a set of metadata, data, and pivot tables. Thus, if you apply traditional performance-tuning techniques based on the data and schema of your organization, you might not see the effect you expect on the actual, underlying data structures.

> **Note:** As a customer, you cannot optimize the SQL underlying many application operations because it is generated by the system, not written by each tenant.

### Search Architecture

Search is the capability to query records based on free-form text. The Salesforce search architecture is based on its own data store, which is optimized for searching for that text.

Salesforce provides search capabilities in many areas of the application, including:

- The sidebar
- Advanced and global searches
- Find boxes and lookup fields
- Suggested Solutions and Knowledge Base
- Web-to-Lead and Web-to-Case
- Duplicate lead processing
- Salesforce Object Search Language (SOSL) for Apex and the API

For data to be searched, it must first be indexed. The indexes are created using the search indexing servers, which also generate and asynchronously process queue entries of newly created or modified data. After a searchable object's record is created or updated, it could take about 15 minutes or more for the updated text to become searchable.

Salesforce performs indexed searches by first searching the indexes for appropriate records, then narrowing down the results based on access permissions, search limits, and other filters. This process creates a result set, which typically contains the most relevant results. After the result set reaches a predetermined size, the remaining records are discarded.

## Infrastructure for Systems with Large Data Volumes

### Lightning Platform Query Optimizer

The Salesforce multitenant architecture uses the underlying database in such a way that the database system's optimizer can't effectively optimize search queries. The Lightning Platform query optimizer helps the database's optimizer produce effective queries by providing efficient data access in Salesforce.

The Lightning Platform query optimizer works on automatically generated queries that handle reports, list views, and SOQL queries. Specifically, the optimizer:

- Determines the best index from which to drive the query, if possible, based on filters in the query
- Determines the best table from which to drive the query, if no good index is available
- Determines how to order the remaining tables to minimize cost
- Injects custom foreign key value tables that are required to create efficient join paths
- Influences the execution plan for the remaining joins, including sharing joins, to minimize database I/O
- Updates statistics

#### Creating Efficient Queries

When you're working with large volumes of data, it's important to build efficient SOQL queries, reports, and list views, which all depend on selectivity and indexes.

**Use SOQL to Determine the Selectivity of a Filter Condition**

Consider a query that uses a basic, unary WHERE clause condition:

```soql
SELECT Id, Name FROM Opportunity
WHERE Stagename = 'Closed Won'
```

Execute the following query to get statistics related to your filter condition's selectivity:

```soql
SELECT Stagename, COUNT(id) FROM Opportunity
GROUP BY ROLLUP (Stagename)
```

The result set shows you the distribution of records for each value of the Stagename picklist field, including the total number of records in the object.

**Determine the Selectivity of More Complex Filter Conditions**

For a query with a date field and the AND operator:

```soql
SELECT Id, Name FROM Opportunity
WHERE Stagename = 'Closed Won'
AND CloseDate = THIS_WEEK
```

Get statistics for the CloseDate field:

```soql
SELECT WEEK_IN_YEAR(CloseDate), CALENDAR_YEAR(CloseDate), COUNT(id)
FROM Opportunity
GROUP BY ROLLUP(WEEK_IN_YEAR(CloseDate),CALENDAR_YEAR(CloseDate))
ORDER BY CALENDAR_YEAR(CloseDate), WEEK_IN_YEAR(CloseDate)
```

For filter conditions that combine two or more conditions (using AND), the query optimizer considers the overall filter condition selective when the filter targets less than:

- Twice the selectivity thresholds for each filter
- The selectivity thresholds for the intersection of those fields

> **Tip:** With the OR operator, each filter must meet the threshold individually.

**Understand the Impact of Deleted Records on Selectivity**

To collect the selectivity statistics for `Opportunity.StageName` and explicitly exclude deleted records:

```soql
SELECT Stagename, COUNT(id) FROM Opportunity WHERE IsDeleted=false GROUP BY Stagename
```

**Make Sure Fields Have Indexes**

The Database Query & Search Optimization Cheat Sheet lists the standard fields that have an index by default, including `Id`, `Name`, `OwnerId`, `CreatedDate`, `SystemModstamp`, and `RecordType`, as well as all master-detail and lookup fields.

If your filter condition involves a custom field, work with Customer Support to create a custom index on the field your filter uses. Not all fields can have an index, such as non-deterministic formula fields.

### Database Statistics

Modern databases gather statistics on the amount and types of data stored inside of them, and they use this information to execute queries efficiently. Because of Salesforce's multitenant approach to software architecture, the platform must keep its own set of statistical information to help the database understand the best way to access the data. This statistics-gathering process runs on a nightly basis.

### Skinny Tables

Salesforce can create skinny tables to contain frequently used fields and to avoid joins. This can improve the performance of certain read-only operations. Skinny tables are kept in sync with their source tables when the source tables are modified.

If you want to use skinny tables, contact Salesforce Customer Support. When enabled, skinny tables are created and used automatically where appropriate.

#### How Skinny Tables Can Improve Performance

For each object table that's visible to you, Salesforce maintains other, separate tables at the database level for standard and custom fields. This separation ordinarily requires a join when a query contains both kinds of fields. A skinny table contains both kinds of fields and also omits soft-deleted records.

Read-only operations that reference only fields in a skinny table don't require an extra join, and can consequently perform better. Skinny tables are most useful with tables containing millions of records to improve the performance of read-only operations, such as reports.

> **Important:** Skinny tables aren't a magic wand to wave at performance problems. There's overhead in maintaining separate tables that hold copies of live data.

Skinny tables can be created on custom objects, and on Account, Contact, Opportunity, Lead, and Case objects.

**Skinny tables can contain the following types of fields:**

- Checkbox
- Currency
- Date
- Date and time
- Email
- Number
- Percent
- Phone
- Picklist (multi-select)
- Text
- Text area
- Text area (long)
- URL

Skinny tables and skinny indexes can also contain encrypted data.

**Considerations:**

- Skinny tables can contain a maximum of 200 columns.
- Skinny tables can't contain fields from other objects.
- For Full sandboxes: Skinny tables are copied to your Full sandbox orgs.
- For other types of sandboxes: Skinny tables aren't copied to your sandbox organizations.

### Indexes

Salesforce supports custom indexes to speed up queries. You can create custom indexes by contacting Salesforce Customer Support or by deploying a custom index XML file via the Metadata API.

> **Note:** The custom indexes that Salesforce Customer Support creates in your production environment are copied to all sandboxes that you create from that production environment.

**The platform maintains indexes on the following fields for most objects:**

- RecordTypeId
- Division
- CreatedDate
- Systemmodstamp (LastModifiedDate)
- Name
- Email (for contacts and leads)
- Foreign key relationships (lookups and master-detail)
- The unique Salesforce record ID (primary key for each object)

Salesforce also supports custom indexes on custom fields, except for text areas (long), text areas (rich), non-deterministic formula fields, and encrypted text fields.

External IDs cause an index to be created on that field.

**You can create External IDs only on the following fields:**

- Auto Number
- Email
- Number
- Text

To create custom indexes for other field types, including standard fields, contact Salesforce Customer Support.

#### Index Tables

The Salesforce multitenant architecture makes the underlying data table for custom fields unsuitable for indexing. To overcome this limitation, the platform creates an index table that contains a copy of the data, along with information about the data types.

By default, the index tables don't include records that are null (records with empty values). You can work with Salesforce Customer Support to create custom indexes that include null rows.

#### Standard and Custom Indexed Fields

The query optimizer maintains a table containing statistics about the distribution of data in each index. It uses this table to perform pre-queries to determine whether using the index can speed up the query.

**Standard Indexed Fields**

Used if the filter matches less than 30% of the first million records and less than 15% of additional records, up to a maximum of one million records.

Examples:
- A query against a table with 2 million records: index used if filter matches 450,000 or fewer records.
- A query against a table with 5 million records: index used if filter matches 900,000 or fewer records.

**Custom Indexed Fields**

Used if the filter matches less than 10% of the total records, up to a maximum of 333,333 records.

Examples:
- A query against a table with 500,000 records: index used if filter matches 50,000 or fewer records.
- A query against a table with 5 million records: index used if filter matches 333,333 or fewer records.

**AND/OR/LIKE behavior:**

- For AND: the query optimizer uses indexes unless one returns more than 20% of the object's records or 666,666 total records.
- For OR: the query optimizer uses indexes unless they all return more than 10% of the object's records or 333,333 total records. All fields in the OR clause must be indexed for any index to be used.
- For LIKE: the query optimizer samples up to 100,000 records of actual data to decide whether to use the custom index.

Custom indexes can be created on deterministic formula fields.

**Non-deterministic formula fields (cannot be indexed) include those that:**

- Reference other entities (like fields accessible through lookup fields)
- Include other formula fields that span other entities
- Use dynamic date and time functions (for example, TODAY and NOW)
- Reference Owner, autonumber, divisions, or audit fields (except CreatedDate and CreatedByID)
- Reference multi-select picklists, currency fields in a multicurrency org, long text area fields, binary fields

**Standard fields with special functionalities (non-deterministic):**

- Opportunity: Amount, TotalOpportunityQuantity, ExpectedRevenue, IsClosed, IsWon
- Case: ClosedDate, IsClosed
- Product: ProductFamily, IsActive, IsArchived
- Solution: Status
- Lead: Status
- Activity: Subject, TaskStatus, TaskPriority

#### Two-Column Custom Indexes

Two-column custom indexes are useful for list views and situations where you want to use one field to select records to display and another field to sort them.

For a query pattern like:

```soql
SELECT Name
FROM Account
WHERE f1__c = 'foo'
AND f2__c = 'bar'
```

A two-column index on `f1__c, f2__c` is more efficient than single indexes on each field.

> **Note:** Two-column indexes can have nulls in the second column, whereas single-column indexes can't unless Salesforce Customer Support explicitly enabled the option.

### Divisions

Divisions are a means of partitioning the data of large deployments to reduce the number of records returned by queries and reports. Divisions are beneficial for organizations with large amounts of data. To use divisions, your organization must have over one million records in a single object and more than 35 licenses.

## Techniques for Optimizing Performance

### Using Mashups

One approach to reducing the amount of data in Salesforce is to maintain large data sets in a different application, and then make that application available to Salesforce as needed. Salesforce supports the following mashup designs:

**External Website:** The Salesforce UI displays an external website, and passes information and requests to it.

**Callouts:** Apex code allows Salesforce to use Web services to exchange information with external systems in real time.

**Advantages:**
- Data is never stale.
- No proprietary method needs to be developed to integrate the two systems.

**Disadvantages:**
- Accessing data takes more time.
- Functionality is reduced. For example, reporting and workflow do not work on the external data.

### Defer Sharing Calculation

In some circumstances, it might be appropriate to use a feature called defer sharing calculation, which allows users to defer the processing of sharing rules until after new users, rules, and other content have been loaded.

An organization's administrator can use a defer sharing calculation permission to suspend and resume sharing calculations, and to manage two processes: group membership calculation and sharing rule calculation.

### Using SOQL and SOSL

| | SOQL | SOSL |
|---|---|---|
| Executes with | `query()` call | `search()` call |
| Uses the | Database | Search indexes |

**Use SOQL when:**
- You know in which objects or fields the data resides.
- You want to retrieve data from a single object or related objects, count records, sort results, or retrieve data from number, date, or checkbox fields.

**Use SOSL when:**
- You don't know in which object or field the data resides.
- You want to retrieve multiple objects and fields efficiently, retrieve data for a particular division using the divisions feature.

**Key considerations:**

- SOSL is generally faster than SOQL if the search expression uses a CONTAINS term.
- SOSL can tokenize multiple terms within a field and builds a search index off this.
- In some cases, when multiple WHERE filters are being used in SOQL, indexes can't be used even though the fields can be indexed. Decompose the single query into multiple queries, each with one WHERE filter, and then combine the results.
- Executing a query with a WHERE filter that has null values for picklists or foreign key fields doesn't use the index.

**Example of a poorly performing query:**

```soql
SELECT Contact__c, Max_Score__c, CategoryName__c, Category__Team_Name__c
FROM Interest__c
WHERE Contact__c != null
AND Contact__c IN :contacts
AND override__c != 0
AND (
    (override__c != null AND override__c > 0)
    OR
    (score__c != null AND score__c > 0)
)
AND Category__c != null
AND (
    (Category_Team_IsActive__c = true OR CategoryName__c IN :selectvalues)
    AND
    (
        Category_Team_Name__c != null
        AND
        Category_Team_Name__c IN :selectTeamValues
    )
)
```

Nulls in the criteria prevented the use of indexes. Rewritten as:

```soql
SELECT Contact__c, Max_Score__c, CategoryName__c, Category_Team_Name__c
FROM Interest__c
WHERE Contact__c IN :contacts
AND (override__c > 0 OR score__c > 0)
AND Category__c != 'Default'
AND (
    (Category_Team_IsActive__c = true OR CategoryName__c IN :selectvalues)
    AND
    Category_Team_Name__c IN :selectTeamValues
)
```

For the field `Category__c`, the value `Default` is substituted for NULL, allowing an index to be used.

**Check for nulls before querying:**

```apex
if (acctid != null) {
    SELECT Name
    FROM Account
    WHERE Account_Id___c = :acctid
}
else {
    return "Not Found"
}
```

### Deleting Data

The Salesforce data deletion mechanism can have a profound effect on the performance of large data volumes. Salesforce uses a Recycle Bin metaphor for data that users delete (soft deletion).

- Deleted items remain in the Recycle Bin for 15 days and can be restored during that time.
- Items in the Recycle Bin don't count against an org's storage usage.
- After 15 days, items are scheduled for hard deletion.
- Bulk API and Bulk API 2.0 support a hard delete option, which allows records to bypass the Recycle Bin.

> **Recommendation:** Use the Bulk API 2.0's hard delete function to delete large data volumes.

### Search

When large volumes of data are added or changed, the search system must index that information before it becomes available for search by all users, and this process might take a long time.

## Best Practices

The main approaches to performance tuning in large Salesforce deployments rely on reducing the number of records that the system must process.

**Approaches for reducing the number of records:**
- Reducing scope by writing queries that are narrow or selective
- Reducing the amount of data kept active (archiving or discarding data at the same rate it enters the system)

### Reporting

| Goal | Best Practice |
|------|---------------|
| Maximizing reporting performance | Partition data to match its likely use; minimize the number of records per object |
| Reducing the number of joins | Minimize the number of objects queried and relationships used in reports; de-normalize data when practical |
| Reducing the amount of data returned | Only add fields to a report, list view, or SOQL query that are required |
| Reducing the number of records to query | Reduce amount of data by archiving unused records; use report filters that emphasize indexed fields |

### Loading Data from the API

| Goal | Best Practice |
|------|---------------|
| Improving performance | Any data operation including more than 2,000 records is a good candidate for Bulk API 2.0; jobs with fewer than 2,000 records should use bulkified synchronous calls in REST (e.g., Composite) or SOAP |
| Using the most efficient operations | `insert()` is fastest, `update()` is next, `upsert()` is next; if possible, break `upsert()` into create() and update() |
| Reducing data to transfer and process | When updating, send only fields that have changed (delta-only loads) |
| Reducing transmission time and interruptions | Authenticate once per load; use GZIP compression and HTTP keep-alive |
| Avoiding unnecessary overhead | Use Public Read/Write security during initial load to avoid sharing calculation overhead |
| Reducing computations | Load users into roles first, then record data with owners, then public groups/queues, then add sharing rules one at a time |
| Deferring computations | Disable Apex triggers, workflow rules, and validations during loads; use batch Apex to process records after load |
| Balancing batch sizes vs. timeouts | When using SOAP API, use as many batches as possible—up to 200—that still avoid network timeouts |
| Minimizing parent record-locking conflicts | When changing child records, group them by parent (`ParentId`) in the same batch |
| Deferring sharing calculations | Use the defer sharing calculation permission to defer sharing calculations until after all data has been loaded |

### Extracting Data from the API

| Goal | Best Practice |
|------|---------------|
| Using the most efficient operations | Use `getUpdated()` and `getDeleted()` SOAP API to sync at intervals greater than 5 minutes; for more frequent syncing use outbound messaging; for queries that can return more than 1 million results consider Bulk API 2.0 |

### Searching

| Goal | Best Practice |
|------|---------------|
| Reducing the number of records to return | Keep searches specific and avoid using wildcards. For example, search with `Michael` instead of `Mi*` |
| Reducing the number of joins | Use single-object searches for greater speed and accuracy |
| Improving efficiency | Use the setup area for searching to enable language optimizations; turn on enhanced lookups and auto-complete |
| Improving search performance | In some cases, partition data with divisions |

### SOQL and SOSL

| Goal | Best Practice |
|------|---------------|
| Allow indexed searches when SOQL queries with multiple WHERE filters can't use indexes | Decompose the query — if two indexed fields joined by OR exceed the index threshold, break into two queries and join results |
| Avoid querying on formula fields | If you must query on formula fields, avoid filtering with non-deterministic references |
| Use the most appropriate language | See Using SOQL and SOSL |
| Execute queries with null values in WHERE for picklists or foreign key fields | Use values such as `NA` to replace NULLs |
| Build efficient SOQL and SOSL queries | Use selective filters, use the Name field when filtering on FirstName and LastName, avoid negative filters, use IN instead of a large list of OR statements, avoid cross-object reference formula fields |
| For SOSL | Use selective filters, don't index custom objects you don't want searched, use specific search terms, use targeted search groups (NAME, EMAIL, PHONE fields) |
| Avoid timeouts on large SOQL queries | Tune the SOQL query, reduce scope, use selective filters, consider Bulk API 2.0, consider adding a LIMIT clause (starting with 100,000 records) |

### Deleting Data

| Goal | Best Practice |
|------|---------------|
| Deleting large volumes of data | Use the hard delete option of Bulk API or Bulk API 2.0; deleting records with many children — delete the children first |

### General

| Goal | Best Practice |
|------|---------------|
| Avoiding sharing computations | Avoid having any user own more than 10,000 records |
| Improving performance | Use a data-tiering strategy that spreads data across multiple objects |
| Reducing time to create full copies of production sandboxes | Exclude field history if not required; don't change a lot of data until the sandbox copy is created |
| Making deployments more efficient | Distribute child records so that no parent has more than 10,000 child records |

## Large Data Volumes Case Studies

### Data Aggregation

**Situation:** A customer needed to aggregate monthly and yearly metrics using standard reports. Monthly details were stored in custom objects with 4 million records and yearly details with 9 million records.

**Solution:** Create an aggregation custom object that summarizes the monthly and yearly values into the required format. Populate the summarization object using batch Apex. Run reports from the aggregated custom object.

### Custom Search Functionality

**Situation:** A customer needed to search across multiple objects using specific values and wildcards. They created a custom Visualforce page that allowed users to enter 1–20 different fields and search using SOQL. When wildcards were introduced, queries took longer. Querying across multiple objects resulted in multiple queries.

**Solutions:**
- Use only essential search fields to reduce the number of searchable fields and allow indexes to be used.
- De-normalize the data from multiple objects into a single custom object to avoid multiple querying calls.
- Dynamically determine the use of SOQL or SOSL based on both the number of fields searched and the types of values entered. Very specific values (no wildcards) used SOQL.

### Indexing with Nulls

**Situation:** A customer needed to allow nulls in a field and be able to query against them. Single-column indexes for picklists and foreign key fields exclude null rows.

**Solution:** The best practice is to not use null values initially. Use some other string, such as `N/A`, in place of NULL. If records already exist with null values, create a formula field that displays text for nulls, then index that formula field.

Example — if the `Status` field is indexed and contains nulls, this query prevents the index from being used:

```soql
SELECT Name
FROM Object
WHERE Status__c = ''
```

Create a formula called `Status_Value__c`:

```
Status_Value__c = IF(ISBLANK(Status__c), "blank", Status__c)
```

Then query using:

```soql
SELECT Name
FROM Object
WHERE Status_Value__c = 'blank'
```

### Rendering Related Lists with Large Data Volumes

**Situation:** A customer had hundreds of thousands of account records and 15 million invoices in a master-detail relationship. Each account record took a long time to display due to the Invoices related list's lengthy rendering time.

**Solution:** The delay was caused by data skew. Reduce the number of invoice records for parent records with thousands of them and keep data skew to a minimum. Use the "Enable Separate Loading of Related Lists" setting to allow the account detail to render while the related list query completes.

### API Performance

**Situation:** A custom integration synchronized Salesforce data with external applications. The integration queried all data in a given object, loaded it into external systems, then queried for IDs of all data to determine what was deleted. The objects contained several million records, and queries were taking minutes to complete. The integration used a specific API user that was part of the sharing hierarchy to limit the records retrieved.

**Solution:** Give the query access to all the data and use selective filters to get the appropriate records. Using an administrator as the API user would provide access to all data and prevent sharing from being considered in the query. Also create a delta extraction to lower the volume of data that needed to be processed.

### Sort Optimization on a Query

**Situation:** A customer had the following query:

```soql
SELECT Id,Product_Code__c
FROM Customer_Product__c
WHERE CreatedDate = Last_N_Days:3
```

The amount of data exceeded the threshold for standard indexes: 30% of the total records up to one million records.

**Solution:** Rewrite the query as:

```soql
SELECT Id,Product_Code__c
FROM Customer_Product__c
WHERE CreatedDate = Last_N_Days:3
ORDER BY CreatedDate LIMIT 99999
```

In this query, the threshold checks were not done, and the `CreatedDate` index was used to find the records.

> **Note:** In general, when querying for data added over `Last_N_Days`, if you specify an ORDER BY query on an indexed field with a limit of fewer than 100,000 records, the ORDER BY index is used to do the query.

### Multi-Join Report Performance

**Situation:** A customer created a report using four related objects: Accounts (314,000), Sales Orders (769,000), Sales Details (2.3 million), and Account Ownership (1.2 million). The report had very little filtering.

**Solution:**
- Add additional filters to make the query more selective and ensure as many filters as possible are indexable.
- Reduce the amount of data in each object, whenever possible.
- Keep the Recycle Bin empty. Data in the Recycle Bin affects query performance.
- Ensure no complex sharing rules exist for the four related objects. Complex sharing rules can have a noticeable impact on performance.

## Summary

The Salesforce platform is a robust environment in which native and custom applications can scale to large volumes of data very quickly while continuing to perform well.

You can maximize the benefits of these capabilities by:

- **Making queries selective** — ensure that reports, list views, and SOQL are using appropriate filters.
- **Reducing the amount of active data** — use archiving, mashups, and other techniques to reduce the amount of data stored in Salesforce.

Following these two broad principles and the best practices that support them can reduce the impact of large data volumes on Salesforce application performance.
