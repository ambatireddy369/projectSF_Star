# LLM Anti-Patterns — Case Management Setup

Common mistakes AI coding assistants make when generating or advising on Case Management Setup.
These patterns help the consuming agent self-check its own output.

## Anti-Pattern 1: Treating Auto-Response Rules as Independent of Assignment Rules

**What the LLM generates:** Instructions to configure an auto-response rule as a standalone mechanism — "create an auto-response rule, set the email template, activate it" — without mentioning the assignment rule dependency.

**Why it happens:** The auto-response rule has its own Setup page, its own activation checkbox, and its own criteria fields. It looks like an independent feature. LLMs trained on general Salesforce documentation surface the configuration steps without capturing the runtime dependency on assignment rule execution.

**Correct pattern:**

```
Correct sequence:
1. Confirm the active case assignment rule exists and fires for the relevant case creation events.
2. Verify at least one rule entry matches (or add a catch-all).
3. Only then configure the auto-response rule.
4. Test by checking Case History for "Rule Assignment" entry — if absent, auto-response will not fire.
```

**Detection hint:** Any response that tells you to configure or debug an auto-response rule without first verifying the assignment rule layer is incomplete. Flag: "auto-response rule" mentioned without "assignment rule" in the same procedure.

## Anti-Pattern 2: Claiming Escalation Rules Fire in Real Time

**What the LLM generates:** Advice that escalation rules will "immediately" escalate a case when the threshold is reached, or that SLAs of 30 minutes or less are achievable with native escalation rules.

**Why it happens:** LLMs infer real-time behavior from the word "rule" — rules sound like reactive triggers. The hourly cadence of the escalation engine is a platform-specific operational detail that does not appear in general automation reasoning.

**Correct pattern:**

```
Escalation rules run on an hourly engine cadence.
A case that crosses the threshold at 9:05 AM will not be escalated
until the engine next runs — potentially at 10:00 AM.
For sub-hour SLAs, use Flow time-based actions or Apex scheduled jobs.
```

**Detection hint:** Any statement like "escalate immediately when the threshold is reached" or "escalation fires in real time" is incorrect for native escalation rules. Flag: "immediately" or "real-time" in escalation context.

## Anti-Pattern 3: Omitting Business Hours from Escalation Rule Entries

**What the LLM generates:** Escalation rule configuration steps that set the time threshold and action but do not mention attaching a business hours record to the rule entry.

**Why it happens:** Business hours is a separate Salesforce object that must be pre-configured and then referenced in the escalation rule entry. LLMs often describe the escalation rule entry fields that are most prominent in the documentation (time trigger, action type, assigned-to user) without noting that the business hours field defaults to blank — meaning 24/7 clock measurement.

**Correct pattern:**

```
Setup → Business Hours → New → configure operating schedule and timezone.
Setup → Escalation Rules → [rule] → Rule Entries → [entry] → Business Hours: [select the record]

Without this attachment, elapsed time is measured in calendar hours (24/7),
not business hours, causing after-hours and weekend escalations.
```

**Detection hint:** Any escalation rule entry configuration that does not include the business hours field assignment. Flag: "escalation rule entry" without "business hours" mentioned.

## Anti-Pattern 4: Assuming Web-to-Case Has Native Field Validation

**What the LLM generates:** Advice to use required fields or picklist constraints in the Web-to-Case setup to enforce data quality on form submissions.

**Why it happens:** Other form builders (Experience Cloud, Flow screens, third-party tools) have native validation. LLMs generalize this capability to Web-to-Case without recognizing that the generated HTML is a plain form snippet with no built-in validation — Salesforce does not reject malformed submissions.

**Correct pattern:**

```
Web-to-Case has no native field validation.
The generated HTML form contains no JavaScript validation.
All validation must be implemented as:
  - Client-side JavaScript in the HTML form (not generated by Salesforce)
  - Post-creation validation via Flow triggered on case creation
  - Apex before-insert trigger on Case

Without custom validation, any value (including empty strings, malformed
email addresses, and spam content) will create a case in Salesforce.
```

**Detection hint:** Any response suggesting Web-to-Case has required-field enforcement or built-in validation on the Salesforce side. Flag: "required" or "validation" in Web-to-Case context without noting it must be custom-implemented.

## Anti-Pattern 5: Suggesting Entitlement Templates on Products Are Available in Lightning

**What the LLM generates:** Instructions to add entitlement templates to product records via the product detail page in Lightning Experience, so that cases created for products auto-receive an entitlement.

**Why it happens:** The Salesforce documentation describes entitlement templates on products as a supported feature. LLMs do not reliably surface the caveat that this configuration UI is only accessible in Salesforce Classic. The feature exists, but the configuration path is Classic-only.

**Correct pattern:**

```
Adding entitlement templates to products is a Classic-only configuration.
In Lightning Experience, there is no equivalent UI on the product record.

To auto-apply entitlements in Lightning:
  - Use a Flow triggered on Case creation that queries the appropriate
    entitlement for the Account/Contact and assigns it to the Case.
  - Use an Apex trigger on Case insert with similar logic.

The entitlement process itself is configurable in Lightning; only the
template-on-product UI path is Classic-only.
```

**Detection hint:** Any response describing a product record entitlement template configuration in Lightning UI. Flag: "entitlement template" + "product" + "Lightning" without the Classic caveat.

## Anti-Pattern 6: Ignoring the Email-to-Case Body Truncation Limit

**What the LLM generates:** Email-to-Case setup instructions that describe the feature without mentioning the 32,000-character body truncation limit.

**Why it happens:** The 25 MB attachment size limit is more visible in the documentation than the body character limit. LLMs surface the headline limit but miss the body truncation, which silently drops content from long technical support emails.

**Correct pattern:**

```
Email-to-Case truncates the email body at 32,000 characters.
Content beyond this limit is permanently lost — it is not stored
as an attachment or surfaced in any error message.

For technical support orgs receiving long log files or detailed
error traces, communicate this limit to the support team. For
content beyond 32,000 characters, agents should request the
information as an attachment rather than inline text.
```

**Detection hint:** Email-to-Case setup instructions that describe body handling without mentioning the 32,000-character truncation limit. Flag: "email body" in Email-to-Case context without the character limit.
