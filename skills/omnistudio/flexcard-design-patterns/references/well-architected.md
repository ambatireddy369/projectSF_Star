# Well-Architected Notes — FlexCard Design Patterns

## Relevant Pillars

### User Experience

FlexCards are a primary UX surface in OmniStudio. Card design directly determines whether users see clear, actionable record information or blank panels and unexplained failures. Well-architected FlexCards:
- Render meaningful content immediately on load without visible delay.
- Show explicit error messages when data is unavailable — never blank.
- Present contextually relevant actions (conditional visibility) rather than disabled or hidden controls that confuse users.
- Use the Saved state to confirm write operations so users do not re-submit.

### Performance

FlexCards are rendered in Lightning App Pages and Experience Cloud pages that may contain multiple components. Each FlexCard's data sources add to page load time. Well-architected FlexCards:
- Use one Integration Procedure data source to replace multiple parallel SOQL or Apex calls.
- Defer child record data retrieval to the parent IP rather than firing per-row child card queries.
- Use Streaming data sources only when real-time updates are genuinely required — not as a default.
- Avoid deeply nested child card hierarchies (more than two levels) which compound render time.

## Architectural Tradeoffs

### Integration Procedure vs. Direct SOQL Data Source

An Integration Procedure adds a dependency and a testable contract. A direct SOQL data source is simpler to configure but does not scale beyond a single object. The tradeoff: use SOQL for genuinely simple, single-object, read-only cards on low-traffic pages. Default to Integration Procedure for anything on a high-traffic record page, anything requiring data from more than one object, and anything that must transform data before rendering.

### Compiled LWC vs. Dynamic Rendering

FlexCards compile to LWC at activation time. This makes them fast at runtime — there is no interpretation overhead. The cost is that the activation step is mandatory after every template change and before every deployment. Teams that treat FlexCards like configuration (always live on change) are caught out by the compile/activate cycle.

### Versioned Releases vs. In-Place Updates

FlexCard versioning allows a new version to be developed and tested while the current version remains in production. This is the correct pattern for high-traffic cards on key record pages. The cost is maintaining two versions during the transition window. For low-risk cosmetic changes on internal pages, in-place updates (increment minor version, activate) are acceptable.

## Anti-Patterns

1. **No Error State Configured** — The single most common FlexCard production failure. A card with no Error state template renders blank when its data source fails. Users report broken pages, support tickets escalate, and the root cause is invisible to the user. Always configure the Error state.

2. **Multiple Independent Data Sources Per Card** — Using two, three, or four SOQL/Apex data sources on a single card to avoid the overhead of building an Integration Procedure. Each source fires independently on card load. On a page with several such cards, this creates a burst of parallel queries that degrades page performance and burns governor limits. Consolidate with an Integration Procedure.

3. **Child Card With Independent Data Source in an Iterated List** — Placing a SOQL or Apex data source on a child card that is iterated by a parent. At 20 related records, this fires 20 queries on a single page render. All child record data must be retrieved in the parent's IP and passed down as a pre-fetched collection.

4. **Deploying Without Activation** — Promoting FlexCard metadata to production via SFDX or a change set without running the activation step in the target environment. The compiled LWC is not redeployed by metadata promotion — it must be regenerated by activating the card in each environment.
