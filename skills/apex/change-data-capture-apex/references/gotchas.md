# Gotchas — Change Data Capture Apex

Non-obvious Salesforce platform behaviors that cause real production problems in this domain.

## Gotcha 1: CDC Trigger Logs Appear Under Automated Process, Not the Current User

**What happens:** A CDC trigger fires after a record is saved, but no debug log appears in the Developer Console's Logs tab for the user who performed the DML. Developers conclude the trigger is not executing and spend time debugging deployment issues.

**When it occurs:** Any time a CDC Apex trigger executes. CDC triggers always run under the **Automated Process** entity, which is a system-level context separate from the current user. Logs generated by the trigger are attached to Automated Process, not the user session open in Developer Console.

**How to avoid:** Before testing a CDC trigger, set up a trace flag specifically for the Automated Process entity:
1. Setup > Debug Logs > New
2. Traced Entity Type: **Automated Process**
3. Set an appropriate time window and debug level.

Logs will then appear in Setup > Debug Logs, not in the Developer Console log tab. Unit tests (which run synchronously) are the exception — CDC trigger execution during tests does appear in test logs.

## Gotcha 2: Object Not Enabled for CDC — Trigger Compiles but Never Fires

**What happens:** A CDC trigger is deployed successfully and shows no errors. DML on the object occurs, but the trigger never fires. No error is raised.

**When it occurs:** When the underlying object has not been enabled in Setup > Integrations > Change Data Capture. The trigger compiles against the change event type (e.g., `AccountChangeEvent`), but no events are ever published to the event bus for untracked objects. The trigger is syntactically valid but dead.

**How to avoid:** Before deploying a CDC trigger to any environment, confirm the object is selected in Setup > Change Data Capture. In CI/CD pipelines, include a pre-deployment validation that queries `SELECT QualifiedApiName FROM EntityDefinition WHERE IsChangeEventEnabled = true` via Tooling API to verify all required entities are tracked.

## Gotcha 3: changedFields Contains Encoded Values in Some API Contexts — But Not in Apex

**What happens:** Developers familiar with Pub/Sub API (gRPC) know that `changedFields` is a base64-encoded bitmap in the binary Avro payload and must be decoded before use. They apply the same assumption in Apex and write decoding logic that corrupts or misreads the values.

**When it occurs:** When a developer ports Pub/Sub API subscriber code to an Apex trigger without accounting for the different runtime context.

**How to avoid:** In Apex CDC triggers, `ChangeEventHeader.changedFields` is already decoded and returned as a `List<String>` of field API names. No decoding is required. Use `header.changedFields.contains('FieldApiName__c')` directly. The base64 encoding is an implementation detail of the wire protocol — the Apex runtime handles deserialization before the trigger receives the event.

## Gotcha 4: One Trigger Per Object — No Multi-Entity Apex Subscriber

**What happens:** An architect designs a CDC listener using a single Apex handler class, intending to route by `header.entityName` to handle Account, Contact, and Opportunity changes in one place. They are surprised to discover that only one trigger file exists and it only receives `AccountChangeEvent` events, not the others.

**When it occurs:** When developers assume CDC Apex triggers can subscribe to multi-entity channels (like `/data/ChangeEvents` in CometD/Pub/Sub API). Apex triggers can only be bound to a single change event type — there is no multi-entity Apex trigger.

**How to avoid:** Create a separate `after insert` trigger for each tracked object. Share business logic via handler classes (e.g., `CDCRouter.dispatch(header, ids)`) that the per-object triggers call. Each trigger file handles one change event type only. If a multi-entity subscriber is required in Apex, use separate triggers that delegate to common handler methods.

## Gotcha 5: DELETE Events — Record Cannot Be Queried Without ALL ROWS

**What happens:** A CDC trigger receives a `DELETE` change event. The trigger attempts to SOQL the deleted record to retrieve its field values for logging or downstream sync. The SOQL returns zero rows and the logic silently drops the delete notification.

**When it occurs:** In the `changeType == 'DELETE'` branch, when the trigger attempts `SELECT Id, Name FROM Account WHERE Id IN :deletedIds`. Deleted records are no longer in the standard SOQL scope.

**How to avoid:** Do not query the record body after a DELETE event. The change event header provides the record IDs — use them to issue a delete notification or cleanup action downstream without fetching field values. If you need the last known field state, it must have been captured during a previous CREATE or UPDATE event and stored externally. If the use case truly requires archived field values after delete, use `SELECT ... FROM Account WHERE Id IN :ids ALL ROWS` — but understand that this returns soft-deleted records in the Recycle Bin only, and not permanently deleted ones.
