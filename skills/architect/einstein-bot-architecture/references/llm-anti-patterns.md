# LLM Anti-Patterns — Einstein Bot Architecture

Common mistakes AI coding assistants make when generating or advising on Einstein Bot Architecture.
These patterns help the consuming agent self-check its own output.

## Anti-Pattern 1: Conflating Einstein Bots and Agentforce Agents

**What the LLM generates:** Advice that mixes Einstein Bot concepts (Dialogs, Dialog Steps, Bot Variables) with Agentforce concepts (Topics, Actions, Instructions) in the same architecture, as if they are interchangeable.

**Why it happens:** Training data contains documentation for both platforms, and the LLM does not track which platform version the org is using. The two frameworks share some terminology but have fundamentally different dialog models.

**Correct pattern:**

```text
WRONG: "Create a Dialog for the Account Services topic and add
        an Apex Action step to the dialog flow."

RIGHT (Einstein Bot): "Create a Dialog named Account_Services.
        Add a Question step to collect the account number, then
        an Action step that invokes the Apex class GetAccountBalance."

RIGHT (Agentforce): "Create a Topic named Account Services with
        the description 'Help customers check balances and recent
        transactions. Requires identity verification.' Add the
        Get_Account_Balance Apex action to the topic."
```

**Detection hint:** Look for "Dialog" and "Topic" used in the same architecture description without explicitly stating which platform each applies to.

## Anti-Pattern 2: Recommending Fewer Than 20 Utterances Per Intent

**What the LLM generates:** An intent training plan with 5-10 example utterances per intent, often generated by the LLM itself rather than sourced from real customer data.

**Why it happens:** LLMs are good at generating paraphrases and assume that a handful of varied examples is sufficient. Salesforce documentation explicitly states a minimum of 20 utterances, and production accuracy requires 50+.

**Correct pattern:**

```text
WRONG: Intent: billing_inquiry
       Utterances (8):
         "What is my bill?"
         "How much do I owe?"
         ...

RIGHT: Intent: billing_inquiry
       Utterances (50+): Sourced from historical Case.Subject and
       Case.Description fields, chat transcripts, and customer
       survey verbatims. LLM-generated paraphrases may supplement
       but must not replace real customer language.
       Minimum: 20 utterances (Salesforce requirement)
       Production target: 50+ utterances
```

**Detection hint:** Count utterances in any intent training plan. Flag if any intent has fewer than 20.

## Anti-Pattern 3: Omitting Agent Availability Check Before Handoff

**What the LLM generates:** A dialog flow that transfers to an agent as the final step without checking whether agents are online or available, and without providing an alternative path.

**Why it happens:** The "happy path" in most training examples shows a clean transfer. LLMs default to the simplest flow and omit error handling for the transfer step.

**Correct pattern:**

```text
WRONG: Step 5: Transfer to Agent (queue: Support)
       [end of dialog]

RIGHT: Step 5: Check Agent Availability (queue: Support)
       Step 5a (agents available): Transfer to Agent with variables:
         - detected_intent → LiveChatTranscript.Intent__c
         - account_number → LiveChatTranscript.Account_Number__c
       Step 5b (no agents): "Our team is currently unavailable.
         I've created case #[CaseNumber] with your details.
         You'll receive an email update within 24 hours."
         → Invoke Flow: Create_Case_From_Bot
```

**Detection hint:** Search for "Transfer to Agent" without a preceding availability check or without an alternative branch for the unavailable case.

## Anti-Pattern 4: Using Generic Topic Descriptions in Agentforce

**What the LLM generates:** Agentforce topic descriptions that are vague or overly broad, such as "Handle customer service requests" or "Help with account issues."

**Why it happens:** LLMs default to generic language when summarizing scope. In Agentforce, the topic description is the primary mechanism for the LLM to decide whether a customer message matches the topic. Vague descriptions cause over-matching.

**Correct pattern:**

```text
WRONG: Topic: Customer Service
       Description: "Help customers with their service needs."

RIGHT: Topic: Billing Inquiries
       Description: "Help customers understand their current bill
       amount, payment due date, recent charges, and payment
       history. Requires customer identity verification before
       disclosing account details. Does NOT handle payment
       processing, refunds, billing disputes, or plan changes."
```

**Detection hint:** Flag topic descriptions that are shorter than 2 sentences, lack a "Does NOT" exclusion clause, or use words like "general," "various," or "any."

## Anti-Pattern 5: Ignoring Channel Constraints in Dialog Design

**What the LLM generates:** A dialog design that uses rich message components (carousels, image cards, quick reply buttons with icons) without accounting for the fact that these render differently or not at all across channels.

**Why it happens:** LLMs generate dialog designs based on the richest available channel (web chat) and do not consider SMS, WhatsApp, or Slack constraints. Training data rarely includes channel-specific degradation behavior.

**Correct pattern:**

```text
WRONG: Step 3: Display product carousel with 5 options
       [same dialog for web chat, SMS, and WhatsApp]

RIGHT: Step 3 (web chat): Display product carousel with 5 options
       Step 3 (SMS/WhatsApp): Display numbered text list:
         "Reply with a number:
          1. Product A
          2. Product B
          3. Product C"
       Architecture note: Design the base dialog for the most
       constrained channel (SMS: text only, 160 char segments).
       Add rich components as progressive enhancements for web chat.
```

**Detection hint:** Look for carousel, card, image, or rich component references in a dialog design that claims to support SMS or WhatsApp without a channel-specific fallback.

## Anti-Pattern 6: Hallucinating Direct SOQL Access in Bot Steps

**What the LLM generates:** Bot dialog steps that directly query Salesforce data using SOQL, as if the bot has native database access within dialog steps.

**Why it happens:** LLMs trained on Apex and SOQL documentation assume that any Salesforce context supports direct queries. Einstein Bot dialog steps and Agentforce topic actions do not support inline SOQL. Data access requires an Apex action or Flow invocation.

**Correct pattern:**

```text
WRONG: Step 2: Query Account WHERE AccountNumber = [collected_value]
       Step 3: Display Account.Name and Account.Balance__c

RIGHT: Step 2: Invoke Apex Action "LookupAccount"
         Input: account_number (from bot variable)
         Output: account_name, account_balance (to bot variables)
       Step 3: Display "Hi {account_name}, your balance is {account_balance}"
```

**Detection hint:** Search for SOQL keywords (SELECT, FROM, WHERE) appearing in bot dialog step descriptions or topic action definitions.
